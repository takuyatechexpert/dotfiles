# 設計原則

## 単一責務 (SRP)
- 1クラス/1関数 = 1つの責務
- 「何をするか」を一言で説明できる粒度に分割する
- 1ファイル = 1つの関心事: スケジューリングと配信ロジック、パースとレンダリングなど異なる関心事は別ファイルに分離する
- 既存ファイルに機能を追加する前に「このファイルの責務は何か」を確認し、責務が異なるなら新規ファイルを作成する

## DRY (Don't Repeat Yourself)
- 同じロジックが3箇所以上に出現したら抽出を検討（Rule of Three）
- 2箇所の重複は許容し、3回目で抽象化する
- **新しい関数を作る前に、既存コードに同じ責務の関数がないか必ず確認する**（`workspaceSymbol` で検索）
- 既存関数と責務が被る場合は、既存を拡張するか呼び出す

## コンポーネント化
- 外部依存（API, DB, ファイルシステム）はインターフェース境界で分離する
- ビジネスロジックは外部依存をインターフェース経由で扱う（具象クラスへの直接依存は作らない）

## 関心の分離
- 入力の受け口（Gateway/Controller）→ ロジック → 出力を明確に分ける
- ログ・メトリクス・認証などの横断関心事はミドルウェアやデコレータに切り出す
- Gateway/Controller は薄く保つ — 入力の受け取り → サービス/ユースケース呼び出し → 出力の返却のみ
- ビジネスロジック（データ変換・フィルタリング・判定・フォーマット）はサービス/ユースケース層に配置する
- 新機能が既存ファイルの責務と異なる場合は、小さくても新規ファイルを作成する

## クリーンアーキテクチャ
- 依存の方向は外側から内側へ（Frameworks → Interface Adapters → Use Cases → Entities）
- 内側の層は外側の層と独立している（外側のクラス・型をインポートしない）
- ユースケース層はフレームワーク非依存に保つ

## DDD (ドメイン駆動設計)
- ドメインモデルにビジネスルールを集約する
- ユビキタス言語 — コード内の命名はドメインの用語に合わせる
- 値オブジェクト・エンティティ・集約の区別を意識する
- リポジトリパターンでデータアクセスを抽象化する

## TDD (テスト駆動開発)

RED → GREEN → REFACTOR のサイクルを守る。テストは仕様の表現として書く。
詳細は `rules/testing.md` を参照。

## 命名
- 名前から意図が読み取れる具体的な語を選ぶ（`data` / `handle` / `manage` のような汎用名より、ドメインの用語を使う）
- bool は `is_` / `has_` / `can_` プレフィックス
- クラス名は名詞、メソッド名は動詞で始める
