# テスト

## TDD サイクル

新規機能の実装時は以下のサイクルに従う：

1. **RED** — 失敗するテストを先に書く
2. **GREEN** — テストを通過する最小限の実装をする
3. **REFACTOR** — テストを維持しながらコードを改善する

各ステップで必ずテストを実行し、期待通りの結果を確認してから次に進む。

## 検証手段（セルフ・ベリフィケーション）

実装・編集後は必ず対応する検証コマンドを実行し、結果を確認してから完了とする。タスク委任時はこの表から該当コマンドを受け入れ基準として明示する。

| 言語・種別 | 検証コマンド |
|---|---|
| PHP (Laravel) | `make test` → `make lint`（Makefile優先、無ければ `vendor/bin/phpunit` → `vendor/bin/phpstan`） |
| TypeScript/Node | `pnpm test` → `pnpm typecheck` → `pnpm lint`（または npm/yarn を環境に合わせる） |
| Python | `pytest` → `ruff check` → `mypy`（プロジェクトの設定に従う） |
| Go | `go test ./...` → `go vet ./...` |
| 共通 | 編集直後に LSP diagnostics を確認しエラーを解消 |
| UI動作確認 | `chrome-devtools` MCP で実際の画面遷移・コンソールログを確認 |

Makefileが存在する場合は `make help` で定義済みターゲットを先に確認する（`rules/makefile.md` 参照）。

## カバレッジ目標

- ユニットテスト: **80% 以上**
- 新規機能: 必ずテストを先に書く
- バグ修正: 再現テストを書いてから修正する

## テストケース設計

以下の観点でテストケースを網羅する：

- **正常系** — 期待される入力と出力
- **異常系** — 不正な入力・例外・エラー条件
- **境界値** — 最小値・最大値・空・null・ゼロ
- **パターン網羅** — `#[DataProvider()]` を活用し、入力パターンを網羅的にテストする

## PHPUnit アトリビュート

プロジェクト標準として以下のアトリビュートを必ず使用する：

- `#[TestDox('日本語でテストの説明')]` — 全テストメソッドに付与し、可読性を確保する
- `#[CoversClass(対象クラス::class)]` — テストクラスに付与し、カバレッジ対象を明示する
- `#[DataProvider('メソッド名')]` — パターン網羅が必要なテストに使用する

## モック方針

- **外部 API**: `Http::fake()` でレスポンスを固定する。`Http::preventStrayRequests()` を基盤として使用し、未登録リクエストを防止する
- **インターフェース**: `$this->createMock()` + `expects()` + `willReturn()` で振る舞いを定義し、`$this->app->instance()` でDIコンテナに登録する
- **データベース**: 実 DB を使う（モック禁止、マイグレーション齟齬の防止）。`RefreshDatabase` トレイトで自動ロールバックする
- **Redis**: `FlushesRedis` トレイトで各テスト前後にクリアする
- **S3**: `FlushesS3` トレイトで操作をクリーンアップする
- **ファイルシステム**: テスト用一時ディレクトリを使用する
- **時刻依存**: Carbon の相対時刻（`Carbon::now()->addHours(1)` 等）で制御する。必要に応じて `Carbon::setTestNow()` でフリーズも可

## テストデータのセットアップ

以下のパターンを用途に応じて使い分ける：

- **Factory パターン** — 外部 API レスポンスの生成には `Tests\Factories\ExternalApi\*` 配下のファクトリを使用する。`state()` メソッドでバリエーションを表現する
- **DataProvider + ヘルパーメソッド** — テストデータ生成用のプライベートメソッド（`generateRequiredFieldsXml()` 等）で XML やリクエストデータを組み立てる
- **テスト内直接セットアップ** — シンプルなケースではテストメソッド内で heredoc 等を使い完結させる

## テストの品質

- テストの説明は `#[TestDox()]` に日本語で「何を・どうしたら・どうなるか」を記述する
- 1テスト1アサーション を基本とする（複数アサーションは関連する検証のみ）
- `assertSame()` を主軸にし、厳密な型・値の一致を検証する
- テスト間の依存・実行順序依存を作らない（`@depends` 不使用）
- テストデータのセットアップは各テスト内で完結させる（共有状態を避ける）

## 既存スキルとの連携

- 仕様ベースのテスト作成は `/spec-driven-test` スキルを使用する
- テストファーストの実装は `/orchestra-of-agents` 内でも TDD サイクルを適用する
